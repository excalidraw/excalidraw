# T05 — Status atom + session atom + auth wrappers

## Description
Implement the jotai state for sync status and auth session, plus thin auth wrappers over supabase-js.

## Files
- `excalidraw-app/data/supabase/syncStatusAtom.ts` — `export type SyncStatus = 'idle'|'syncing'|'synced'|'error'|'offline'`; `export type SyncStatusState = { status: SyncStatus; lastSyncedAt: number | null; error: string | null }`; an atom initialized to `{ status: 'idle', lastSyncedAt: null, error: null }`.
- `excalidraw-app/data/supabase/sessionAtom.ts` — `sessionAtom` (`Session | null`), derived `userIdAtom` (`string | null`), and `useInitSupabaseSession()` hook that on mount calls `getSession()` to seed the atom and subscribes via `onAuthStateChange`, unsubscribing on cleanup. Use the app's jotai store (`appJotaiStore` from `app-jotai.ts`) consistently.
- `excalidraw-app/data/supabase/auth.ts` — `signInWithMagicLink(email): Promise<{ error: AuthError | null }>` (wraps `supabase.auth.signInWithOtp({ email })`), `signOut()`, `getSession()`, `onAuthStateChange(cb)` returning the subscription handle. No-op/safe when `getSupabaseClient()` is null.

## Read first
- `04-lld-part1-data.md` §B (auth.ts, sessionAtom.ts, syncStatusAtom.ts).
- `04-lld.md` NORMATIVE #M1 (auth return shapes) and #5 (`signInWithMagicLink` is the canonical name).
- Real `excalidraw-app/app-jotai.ts` (how the store/atoms are created) and an existing atom usage for style.

## Acceptance criteria
- `signInWithMagicLink('x@y.com')` calls `auth.signInWithOtp({ email: 'x@y.com' })` and surfaces `{ error }`.
- `useInitSupabaseSession()` seeds `sessionAtom` from `getSession()` and updates it on auth changes; cleans up the subscription.
- `userIdAtom` derives the user id from the session (or null).
- Everything is null-safe when Supabase isn't configured.

### Jotai integration approach (the SAME store the app uses)
- `atom` is imported from `../../app-jotai` (NOT directly from `jotai`) in both atom files — this is the re-export bound to the app's single `appJotaiStore` (`createStore()` in `app-jotai.ts`), matching the data-layer convention used by `LocalData.ts` (`import { appJotaiStore, atom } from "../app-jotai"`) and `Collab.tsx`.
- The `useInitSupabaseSession` hook uses `useSetAtom` from `../../app-jotai` (same convention as `app-language/LanguageList.tsx`) plus `useEffect` from `react`. Because the app wraps the tree in `<Provider store={appJotaiStore}>`, the standard `useSetAtom`/`useAtomValue` hooks resolve to that store automatically — no per-call store arg needed.
- Hook lifecycle: `useEffect(..., [])` seeds the atom via `getSession().then(...)` (guarded by a `mounted` flag), subscribes via `onAuthStateChange((session) => setSession(session))`, and returns a cleanup that flips `mounted = false` and calls the unsubscribe fn (which calls `data.subscription.unsubscribe()`).

### Auth wrapper details (verified real supabase-js v2.45.4 signatures)
- `signInWithOtp(credentials: SignInWithPasswordlessCredentials): Promise<AuthOtpResponse>` — `AuthOtpResponse` carries `{ error: AuthError | null }`; `{ email, options: { emailRedirectTo } }` is a valid credential shape (`lib/types.d.ts:453`). Returns only `{ error }`.
- `signOut(): Promise<{ error: AuthError | null }>` — returned directly.
- `getSession(): Promise<{ data: { session: Session | null }, error }>` — returns `data.session ?? null`.
- `onAuthStateChange(cb): { data: { subscription: Subscription } }`, cb = `(event, session) => void | Promise<void>` — we wrap to `(_event, session) => cb(session)` and return `() => data.subscription.unsubscribe()`.
- Types imported from `@supabase/supabase-js` (`AuthError`, `Session`), which re-exports them via `export * from "@supabase/auth-js"`.

### Deviations
- **Null-client branch of `signInWithMagicLink` returns `{ error: null }`** instead of the synthetic `AuthError` suggested in `04-lld-part1-data.md` §B.3. Rationale: the T05 prompt mandates "return a benign result, do not throw," and constructing/importing the `AuthError` class as a *value* is avoided (it's imported as a type only). The function signature `{ error: AuthError | null }` is unchanged; callers see a benign no-op when sync is disabled. No real email is sent because there is no client.
- `userIdAtom` uses `?.user?.id` (extra optional-chaining per the T05 prompt) rather than the doc's `?.user.id`. Functionally identical, slightly safer.
- No `signInWithEmail` alias added (the canonical name `signInWithMagicLink` is what the UI imports per `04-lld.md` reconciliation #5); the optional alias was explicitly "if the UI agent prefers" and is unused.
- Did NOT modify `client.ts`, `app-jotai.ts`, `app_constants.ts`, or `package.json` (per constraints). No optional test added (kept scope tight; covered indirectly by T06/T07).
