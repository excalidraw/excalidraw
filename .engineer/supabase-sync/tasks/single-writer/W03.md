# W03 ‚Äî lockAtom + hook (poll, role, viewMode, takeOver)

## Description
Add the `lockAtom` (role + holder/presence) and wire the lock lifecycle into `useSupabaseSync`: provide the
new engine deps (sessionId, setLock), run the reader poll loop, derive `viewModeEnabled` + `role`, and expose a
`takeOver()` action. Release the lock on unload.

## Implement (per ¬ß5 "useSupabaseSync.ts", ¬ß2 reader poll)
- **Session id:** mint once per tab, store in `sessionStorage` (so two tabs differ); pass `getSessionId` to the
  engine. (A module-level cached value backed by sessionStorage.)
- **setLock:** `useSetAtom(lockAtom)` ‚Üí pass to engine deps so the engine drives role/holder into the atom.
- **Reader poll loop:** when role==="reader" (and signed in + flag on), poll `readLockState` every POLL_MS=4000:
  update lockAtom; if the version advanced, pull + `applyRemoteScene` (reuse existing applyRemoteScene); if the
  lock became free/expired, the user can claim via `takeOver()` (do NOT auto-claim a fresh reader per decision
  #1 ‚Äî but the design's "first session with no writer auto-becomes writer" happens in engine.start; the poll
  path is for an already-open reader). Follow the design precisely.
- **viewModeEnabled:** derive `const viewModeEnabled = role === "reader"` and return it (App.tsx applies it).
- **takeOver():** expose an async action: call `requestTakeover` (writes the request column). The reader stays
  reader and keeps polling; when the poll sees the lock freed/assigned to this session, it claims (engine or a
  repo `claimLock`) and becomes writer + pulls the flushed scene. Handle the crashed-writer fast path (lock not
  live ‚Üí claim immediately) and the ack-grace timeout per ¬ß4.
- **Release on unload:** on beforeunload/visibilitychange-hidden/unmount, if writer, best-effort release (the
  engine's stop/dispose already does this ‚Äî ensure the hook calls it).
- Return shape extends to `{ status, lastSyncedAt, syncNow, notifyChange, role, viewModeEnabled, takeOver,
  lock }` (or expose lock via the atom). Match the design.

### lockAtom shape
`LockState extends SyncLockState` (the engine's exported `{role, writerId, writerSessionId, lockLive,
takeoverRequested}`) **plus** two hook-owned UI bits: `holderIsMe` (derived `writerSessionId ===
getSessionId()`) and `takeoverInFlight`. `SyncRole` is re-exported from `./syncEngine` (single
canonical type, no duplication). Default = reader/idle (`role:"reader", lockLive:false`, everything
null/false) ‚Äî the safe default matching the engine's own `role="reader"`. Uses `../../app-jotai`
`atom`.

### Session-id mechanism
`getSessionId()` in `lockAtom.ts`: module-level `cachedSessionId`, backed by **`sessionStorage`**
under `STORAGE_KEYS.SESSION_STORAGE_SUPABASE_SESSION_ID` (per-tab, so two tabs differ; a duplicated
tab gets a fresh value). Lazily mints via `crypto.randomUUID()` (guarded for envs without crypto ‚Üí
random `sess-‚Ä¶` fallback) and also falls back to an in-memory cached id if `sessionStorage` is
unavailable (private mode). Passed to the engine as the `getSessionId` dep.

### Engine seam (the become-writer mechanism) ‚Äî small public additions to syncEngine.ts
W02's engine already owned the full writer-side lifecycle (claim-on-start, heartbeat, handoff,
becomeWriter/becomeReader, releaseIfWriter) but exposed **no** reader poll and **no** public way to
promote a reader-engine after a takeover. Per the task's sanctioned approach I added four tiny public
methods that all reuse existing private machinery (heartbeat/handoff/role transitions unchanged):
- `getRole(): SyncRole` ‚Äî lets the hook gate its poll (only a reader polls) + short-circuit takeOver.
- `getLocalVersion(): number | null` ‚Äî exposes `localMeta.version` for the poll's version compare.
- `pullLatest(): Promise<void>` ‚Äî pulls + applies via the existing `repullAndReconcile`/`applyCloud`
  path (no duplication); used by the poll on a version bump and by `tryClaim` on promotion.
- `tryClaim(): Promise<boolean>` ‚Äî the become-writer seam: runs `claimLock`; on `acquired` it pulls
  the previous writer's flushed scene if newer (¬ß4 step 8 / ¬ß6.5) then calls the existing private
  `becomeWriter` (arms heartbeat, pushes lockAtom ‚Üí role writer); on a lost claim ‚Üí `becomeReader`;
  on RPC error ‚Üí stays reader + surfaces offline/error.
- `takeOver(): Promise<boolean>` ‚Äî reader-only; calls `requestTakeover`, returns `immediatelyClaimable`
  (the crashed/gone-writer fast path). Does NOT claim itself (the hook owns the timers/poll, per the
  design's "do NOT duplicate poll logic"). The engine's 22 W02 tests all stay green.

Also switched the engine's local `HEARTBEAT_MS` to import `SUPABASE_LOCK_HEARTBEAT_MS` from
app_constants (now that W03 owns that file).

### Reader poll loop (hook)
A `useEffect` gated on `flag on + client + signed in + lock.role === "reader"`, interval
`SUPABASE_LOCK_POLL_MS` (4000). Each tick (and re-checking `engine.getRole() === "reader"` live):
1. `readLockState(client)` ‚Üí mirror holder/liveness into lockAtom (role stays reader ‚Äî only
   `start()`/`tryClaim()` flip role to writer, per the task);
2. if `state.version > engine.getLocalVersion()` ‚Üí `engine.pullLatest()` (reuses applyRemoteScene);
3. drive a pending takeover: lock free/expired ‚Üí `engine.tryClaim()`; ACK_GRACE elapsed but lease
   still live ‚Üí give up (clear `takeoverInFlight`; never steal a live lease ‚Äî ¬ß4 timeout terminal).
A failed poll RPC is swallowed (keep last-known role/scene, retry next tick). The effect re-arms on
`lock.role` so a flip to writer tears the poll down (a writer heartbeats inside the engine instead).

### takeOver() flow (hook)
`takeOver()` sets `takeoverInFlight=true` + arms `takeoverPendingRef` and a `Date.now()+ACK_GRACE`
deadline ref, then `engine.takeOver()` (writes the request column). If `immediatelyClaimable` ‚Üí
`engine.tryClaim()` at once (crashed/gone writer); otherwise the poll claims the moment the writer
releases / its lease lapses. A won claim flips role‚Üíwriter in the engine, whose `setLock` push (via
the hook's adapter) clears `takeoverInFlight`.

### viewModeEnabled wiring
`const viewModeEnabled = isSupabaseSyncEnabled() && !!userId && lock.role === "reader"`. Returned
from the hook. Flag OFF ‚áí `false` (engine never built, atom stays reader default) ‚áí app identical to
today. The setLock adapter computes `holderIsMe` and preserves/clears `takeoverInFlight` (cleared the
moment role becomes writer). On engine teardown the construction-effect cleanup resets lockAtom to
the reader default so a torn-down engine never strands the UI in view-only.

### Deviations
- The design (¬ß5) places the reader poll inside the engine; the **task spec (W03) explicitly puts the
  poll in the hook** ("Reader poll loop: a `useEffect`‚Ä¶"), and W02's engine shipped without a poll, so
  the poll lives in the hook as the task directs. The hook drives the poll/claim and the engine
  exposes the minimal `getRole`/`getLocalVersion`/`pullLatest`/`tryClaim`/`takeOver` seam it needs ‚Äî
  no poll-logic duplication. Role transitions remain engine-owned (becomeWriter/becomeReader). The
  `getSessionId`/`setLock` deps were left OPTIONAL on `SyncEngineDeps` (the hook always provides them);
  not tightened to required, per the task's "simplest is to just provide them and leave them optional."
