# W02 — Engine: heartbeat, lock claim, reader-gate, graceful handoff

## Description
Extend `SyncEngine` with the writer lock lifecycle: claim on start, heartbeat renew, reader-gating of all
write paths, takeover detection + graceful handoff (flush-success gated), and release on stop/dispose.

## Files
- `excalidraw-app/data/supabase/syncEngine.ts` (+ `SyncEngineDeps`)
- `excalidraw-app/tests/supabase/syncEngine.test.ts` (extend)

## Implement (per `07-single-writer-design.md` §3, §4, §5 "syncEngine.ts")
- **Deps:** add to `SyncEngineDeps`: `getSessionId: () => string` and `setLock: (state) => void` (role + holder
  info → lockAtom, wired by hook in W03). Keep existing deps.
- **start(userId):** after the initial `pullBoard`, attempt `claimLock`. If acquired → role=writer (start
  heartbeat timer at HEARTBEAT_MS=5000). If not acquired → role=reader (do NOT push; the hook runs the reader
  poll). Set lock state via `setLock`.
- **Heartbeat (writer):** every 5s call `renewLock`. STEP A semantics handled in SQL (M2). If renew reports the
  writer no longer holds the lock (`still_writer=false`) → demote to reader (stop heartbeat, setLock reader). If
  renew sees a `takeover_requested_by` (other session) → run the **handoff** (below). Distinguish a network
  failure (don't demote, retry next tick) from a real loss (demote) — per review m? / §3.
- **Reader-gate (M-critical):** as the FIRST line of `notifyChange()`, `flush()`, `syncNow()`, and `runPush()`:
  if role !== "writer", return (no-op; flush returns `true`/resolved so callers don't hang). Per §3 "Where the
  push pipeline is GATED".
- **Flush-success contract (M1):** `flush()` and `runPush()` return `Promise<boolean>` — `true` only when the
  push committed and `dirty===false` afterward. (Existing callers that ignored the return still work.)
- **Handoff (writer sees takeover, §4):** on detecting `takeover_requested_by !== my session`: call
  `flush()`; if it returns **true** → `releaseLock` → demote to reader (setLock reader). If flush returns
  **false** → DO NOT release, stay writer+dirty, surface error/offline status, leave the request set so it
  retries next heartbeat (the requester's lease-expiry path eventually steals it only on true partition).
- **stop()/dispose():** if writer, best-effort `releaseLock` + stop heartbeat. (Hook also calls release on
  unload.)
- Use server-clock liveness only (never `Date.now()` for lease decisions).

### Reader-gate locations (the FIRST statement of each path)
- `notifyChange()` syncEngine.ts:**408** — `if (this.role !== "writer") return;`
- `syncNow()` syncEngine.ts:**446** — `if (this.role !== "writer") return;`
- `flush()` syncEngine.ts:**463** — `if (this.role !== "writer") return true;` (benign — a reader is

- `runPush()` syncEngine.ts:**535** — belt-and-suspenders `if (this.role !== "writer") return true;`.
  - Convention: a reader gate returns `true` (benign no-op), matching "flush success = committed AND
    not dirty" — a reader has nothing to commit, so it is trivially in sync.

### flush() return-type change (M1 flush-success contract)
- `flush(): Promise<boolean>` and the real pipeline now return `boolean`. Refactor: the push pipeline
  body moved to a private **ungated** `pushScene(): Promise<boolean>`; `runPush()` is the gated entry
  (`role check → pushScene()`); `flush()` calls `pushScene()` (already gated at its own top).
  `pushScene()` returns `true` ONLY on the success branch (after `dirty=false` + `synced`); `false`
  on every failure/no-commit branch (signed out, file upload throw, current-scene image errored,
  version conflict→repull, thrown push error, first-insert uniqueness race). `flush()` returns `true`
  for the not-dirty / reader / in-flight-coalesce cases and propagates `pushScene()`'s result
  otherwise (and re-awaits the coalesced rerun's real result). `syncNow()` stays `Promise<void>`.
- The start() first-login INSERT calls `pushScene()` directly (engine is still reader pre-claim, so
  the gated paths would no-op) — exactly one row-establishing push BEFORE the claim (design §6.8).

### start() / claim
- `start()` runs the existing pull+reconcile (extracted to `pullAndReconcile()`), then — only if it
  reconciled cleanly — `claimWriterLock()`. Won claim → `becomeWriter()` (role=writer, arm heartbeat,
  setLock writer/lockLive). Lost claim → `becomeReader(holder…)` (role=reader, setLock with holder).
  Failed claim RPC → stays reader + surfaces offline/error (no optimistic promote).

### Heartbeat (writer, every HEARTBEAT_MS) — `heartbeat()` syncEngine.ts:771
- `renewLock` **throws** (network) → do NOT demote, return, retry next tick.
- `stillWriter === false` → real loss → `becomeReader()` (stops heartbeat).
- live `takeoverRequestedBy` (≠ our session) → `performHandoff()`.
- else → still writer; refresh `setLock` (carries the takeoverRequested flag for UI).

### Handoff — `performHandoff()` syncEngine.ts:822 (M1)
- SUCCESS branch: `const ok = await this.flush()` (runs while role==="writer", gate passes); if
  `ok===true` → `releaseLock` → `becomeReader()` (demote). 
- FAILED-FLUSH branch: `ok===false` → return WITHOUT release/demote — stay writer + dirty; the failed
  push already set error/offline; the next heartbeat re-enters and retries. (Also: if `releaseLock`
  itself throws, stays writer and retries — the un-renewable lease decays as the safety net.)

### stop()/dispose()
- `stop()`: `releaseIfWriter()` (best-effort await/catch) → clear timer + heartbeat → role=reader →
  existing reset. `dispose()`: same release + clear timer + heartbeat + remove window listeners.

### Tests added (10 new; all 12 original kept, adapted to new deps/return types, none weakened)
- writer path: won claim → writer + heartbeat fires renewLock (×2 over 2 ticks), setLock writer.
- reader path: lost claim → reader; notifyChange + advance ⇒ NO pushBoard/saveFiles; flush()⇒true; no
  heartbeat armed.
- reader-gate: syncNow() no-op for reader.
- flush-success: clean push ⇒ true + dirty cleared (follow-up flush no-ops); file-error ⇒ false +
  stays dirty (retry⇒true); network throw on row write ⇒ false + offline, retry ⇒ true.
- handoff success: pending takeover + clean flush ⇒ pushBoard called, releaseLock×1, demoted reader.
- handoff failed-flush (M1): failing flush ⇒ releaseLock NOT called, stays writer; next heartbeat
  (connectivity restored) ⇒ releaseLock×1, demoted reader.
- lost lock: renew stillWriter:false ⇒ reader + heartbeat stops (no further renew).
- heartbeat network throw ⇒ stays writer, keeps heartbeating.
- stop(): writer releases lock + heartbeat stops.
- Test harness: `createHarness` wires `getSessionId` (fixed `SESSION_ID`) + `setLock` (vi.fn);
  boardRepository lock fns mocked via the existing `vi.mock`; `asWriter` (default true) force-sets the
