**Reconcile-after-conflict choice:** on a `{ok:false,conflict:true}` push (or a `23505` first-insert
race throw), the engine re-pulls and applies the cloud row via `applyRemoteScene`, sets
`localMeta = {version: cloud.version, ...}`, re-baselines `lastSyncedSnapshot` to the applied cloud
scene, persists meta, and sets status `synced`. It deliberately leaves `dirty` UNTOUCHED — it does
NOT immediately re-push. If local edits existed, the next `notifyChange()` re-pushes them against the
now-current version (the dirty-check sees the local scene differs from the freshly-applied cloud
baseline). This avoids both a clobber and a recursive re-push loop. (This is the spec's "keep it
simple" option.)

**Deviations honored from the dependency reality:**
- `PushResult = { ok:true; version:number } | { ok:false; conflict:true }` (no `updatedAt`).
- First push uses `expectedVersion === null`; repo INSERTs `version=1` and returns `{ok:true,version:1}`.
- `pushBoard` THROWS on insert errors (no internal `23505`→conflict mapping). The engine wraps the
  first-insert push in try/catch: a uniqueness/race throw (`isUniquenessRace`: code `23505` or
  duplicate-key message) when `expectedVersion === null` triggers repull+reconcile; any other throw
  surfaces `error`/`offline`. A normal guarded-update conflict arrives as `{ok:false,conflict:true}`
  (no throw) and also triggers repull+reconcile.
- Used plain `as const` only (no `as const satisfies` — prettier 2.6.2 parser rejects it). The engine
  has no `as const` need; the test uses none either.
